# Switching from Freeform Classic

:::warning
#### IMPORTANT
For the purpose of clarity in this documentation resource, any prior version of _Freeform (free)_ and _Freeform Pro _2.x_, _3.x_ 4.x_, _5.x_ and _6.x_ are referred to as <span className="text-amber-600 font-bold italic">Freeform Classic</span>. The **new** version of _Freeform_ is referred to as <span className="text-emerald-700 font-bold italic">Freeform Next</span>.
:::

Freeform has been completely rewritten from ground up. Whether you're considering switching from <span className="text-amber-600 font-bold italic">Freeform Classic</span>
to <span className="text-emerald-600 font-bold italic">Freeform Next</span>, or interesting in using Freeform <span className="text-emerald-600 font-semibold italic">Next</span> on your next site build, this page is a resource to
help you understand the differences and changes made with Freeform <span className="text-emerald-600 font-semibold italic">Next</span>.

## How to Upgrade

You must have the appropriate EE-compatible version of Freeform <span className="text-amber-600 font-semibold italic">Classic</span> installed and upgraded for the new
Freeform <span className="text-emerald-600 font-semibold italic">Next</span> to properly detect and migrate it. For example, if you moved from EE2 to EE4, but only have
the EE2 version of Freeform <span className="text-amber-600 font-semibold italic">Classic</span> (4.x) installed, the migration won't work correctly. You'll need to
update to the 5.x (EE3) or 6.x (EE4/EE5) version of Freeform <span className="text-amber-600 font-semibold italic">Classic</span> first. If you're switching from
<span className="text-amber-600 font-semibold italic">Classic</span> to the new Freeform <span className="text-emerald-600 font-semibold italic">Next</span>, we don't expect
you to pay an upgrade fee to access the EE3 or EE4 compatible versions of <span className="text-amber-600 font-semibold italic">Classic</span> to only use it for 15 mins
to run the migration utility, so feel free to temporarily "borrow and reuse" a newer <span className="text-amber-600 font-semibold italic">Classic</span> license from
another site if you have, or [contact us for a temporary copy](/support/).

## Migration Steps

The instructions below will all assume you are ultimately updating to Freeform <span className="text-emerald-600 font-semibold italic">Next</span> 3.x for ExpressionEngine 7.x.

Ultimately, if you're on ExpressionEngine 1.x or 2.x, you'll need to get your site to a minimum of ExpressionEngine 3+ and Freeform
<span className="text-amber-600 font-semibold italic">Classic</span> 5+ in order to make the crossover from
<span className="text-amber-600 font-semibold italic">Classic</span> to <span className="text-emerald-600 font-semibold italic">Next</span>.

## Overview of Key Changes

### Changes to Composer

Freeform <span className="text-emerald-600 font-semibold italic">Next</span> centers itself around the idea of letting admins and/or clients enjoy the experience of building and managing simple or complex forms in an intuitive interface that lets you see a live preview of the forms you're building. We call this [Composer](../forms/forms-composer.mdx), where almost everything is at your fingertips as it aims to stay out of your way and let you do as much as possible without having to move around to other areas in the control panel. At the same time, Freeform <span className="text-emerald-600 font-semibold italic">Next</span> is very powerful and flexible, so there is also a wide variety of other features, settings and options.

For Freeform Next, we now only include a single method for building forms. Freeform <span className="text-amber-600 font-semibold italic">Classic</span> used to allow **EE Template** and **Composer** methods. The very manual EE Template-based approach is no longer a choice. We strongly believe you will be so impressed with the power and flexibility of the new form builder interface, that you won't want to go back to EE Template approach. And even if you do, [you still can build out your forms completely manually](../templating/form.mdx#manual-example) if you desire (but you'll still need to quickly assemble a form layout inside Composer).

[Formatting options inside templates](../templating/form.mdx) or [Formatting Templates](../guides/templating/formatting-templates.mdx) are so flexible, powerful, and easy to use. We carefully came up with a new, intuitive and consistent syntax for templating.

### Changes to Composer Templates

Composer Templates are now referred to as **Formatting Templates** in Freeform <span className="text-emerald-600 font-semibold italic">Next</span>.

While Freeform <span className="text-emerald-600 font-semibold italic">Next</span> offers a very intuitive Composer interface to give you a live preview of the form you're building, there of course isn't exactly a magic way to do this on the front end for your templates. However, we have created an automated way for Freeform to figure out as much of this as possible for you.

As mentioned above, [Formatting options inside templates](../templating/form.mdx) or [Formatting Templates](../guides/templating/formatting-templates.mdx) are so flexible, powerful, and easy to use. We carefully came up with a new, intuitive and consistent syntax for templating.

Formatting Templates are also now stored as HTML files. The benefit here is that they can work nicely with version control and/or staging environments, etc, rather than being stored in the database.

Forms can be generated on the front end templates 2 different ways. There is no worse or better way, but here's an overview:

1. With the [Freeform_Next:Form](../templating/form.mdx) template tag.
   - Your form formatting code is contain directly within the template that you want the form to appear in.
   - No matter what formatting template your form may have assigned to it in Composer, the form always conforms to the template formatting used in this template.
2. Using [Freeform_Next:Render](../templating/form.mdx#render-examples) method.
   - Your form formatting code is stored in a separate template, but is very portable / DRY, and works similar to an include.
     - Formatting templates are HTML files stored in the EE Templates directory (benefit is that they can work nicely with version control and/or staging environments, etc).
     - Freeform includes several [formatting template examples](../guides/templating/formatting-template-examples.mdx) for you to start out with as well.
   - In template(s) that you want your form(s) to show up in, you simply just insert 1 line of code: `{exp:freeform_next:render form="contact"}`

Formatting templates are optional, and only necessary if using the [Freeform_Next:Render tag method](../templating/form.mdx#render-examples), which essentially allows you to attach a formatting template to a form so that you don't need to include formatting inside the template(s) you place the form inside.

The code and formatting in these templates looks exactly like the code in a [Freeform_Next:Form](../templating/form.mdx) tag, but excludes the opening and closing `{exp:freeform_next:form}` tags, as the _Render_ tag does this for you.

### Changes to Freeform Entries

Freeform Entries are now referred to as [**Submissions**](../templating/submissions.mdx) in Freeform <span className="text-emerald-600 font-semibold italic">Next</span>. You'll now use the `{exp:freeform_next:submissions}` template tag to display an individual submission or list of submissions based on some or no criteria.

Freeform <span className="text-emerald-600 font-semibold italic">Next</span> also now includes an option to NOT store submissions in the database, per form.

Freeform <span className="text-emerald-600 font-semibold italic">Next</span> currently only supports basic CSV exporting. In the future, more exporting options will be added.

### Changes to Email Notifications

[Email Notification templates](../forms/email-notifications.mdx) work generally the same way, but now include the option of storing them as HTML files. The benefit here is that they can work nicely with version control and/or staging environments, etc, rather than being stored in the database.

### Changes to Multi-Page Forms

Multi-page forms still work similar to how they did with Freeform Classic, with the exception that there is no unique URI segment per page at this time, and users cannot jump ahead to different pages, only forward and backward 1 page.

### Changes to Spam Protection

One of the greatest shortcomings in Freeform <span className="text-amber-600 font-semibold italic">Classic</span> was spam control. It used to be that you had to either live with out of control spam submissions, or purchase and install another add-on to combat the issue. This is why Freeform <span className="text-emerald-600 font-semibold italic">Next</span> no longer has _Ban Keywords_ feature and does NOT use [EE's native CAPTCHA feature](https://docs.expressionengine.com/latest/security/captchas.html) as it has always proven ineffective. Instead, it includes its own [Javascript-based honeypot spam protection](../forms/spam-protection.mdx), which is immeasurably more effective (and we've had excellent success with it in Freeform for Craft CMS). This is enabled by default, but can be disabled in the [Freeform Settings](../configuration/settings.mdx#spam-protection).

### Changes to Permissions

Freeform <span className="text-emerald-600 font-semibold italic">Next</span> currently does not have any module-level permissions like Freeform <span className="text-amber-600 font-semibold italic">Next</span>. However, this is a feature that will likely be added back in a future version.

### Addition of API Integrations

Freeform <span className="text-emerald-600 font-semibold italic">Next</span> includes integrations with several popular [Mailing List](../integrations/mailing/index.mdx) and [CRM (Customer Relationship Management)](../integrations/crm/index.mdx) API's, including [MailChimp](../integrations/mailing/mailchimp.mdx), [Constant Contact](../integrations/mailing/campaign-monitor.mdx), [Campaign Monitor](../integrations/mailing/constant-contact.mdx), [Salesforce](../integrations/crm/salesforce-lead.mdx) and [HubSpot](../integrations/crm/hubspot.mdx). We plan to add support for other API's in the future as well.

## Migrating from Freeform Classic

Once you have installed Freeform <span className="text-emerald-600 font-semibold italic">Next</span>, don't make any modifications to it, and then review the [Migration utility documentation](./classic-migration.mdx) before running the migration utility. Freeform <span className="text-emerald-600 font-semibold italic">Next</span> can be used alongside Freeform <span className="text-amber-600 font-semibold italic">Classic</span> installations if you're wanting to try out and/or switch to Freeform <span className="text-emerald-600 font-semibold italic">Next</span>.
